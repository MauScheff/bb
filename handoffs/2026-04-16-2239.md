## What is not working

- Current blocker:
  - the lock-screen talk button on the receiving device appears, but transmit from that lock-screen path does not work.
- Strong hypothesis:
  - Apple’s lock-screen talk affordance may not be driving exactly the same app-side begin-transmit path as the foreground hold button, or it may arrive on a different callback ordering.
  - If so, backend lease establishment, renew startup, media-session startup, or local state transitions may not be happening in the expected sequence.
- Important distinction:
  - this seems different from the older lease-loss bug and different from the later Blake foreground-prewarm regression.
  - wake/background receive now works well enough that the lock-screen transmit path can be isolated as its own problem.

## Recommended next step

1. Reproduce specifically from the lock-screen talk button and immediately rerun merged diagnostics.
2. Compare the lock-screen transmit attempt against a known-good foreground hold attempt on the same devices, focusing on:
   - `Requesting system transmit handoff`
   - `System transmit began`
   - `PTT audio session activated`
   - backend `beginTransmit` / lease grant / renew start
   - media session start and first outbound audio
3. If the lock-screen path skips backend begin or media activation, make that path explicit in the state machine instead of assuming foreground hold semantics.
