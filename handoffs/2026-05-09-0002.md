# Handoff 2026-05-09 00:02

## Summary

This handoff records the current call-screen text hierarchy decision while telemetry is being wired into the production call surface.

## Current truth

- The call screen should keep the person as the main object of attention.
- The preferred text order in the identity block is:
  1. display name
  2. call status
  3. telemetry and action-relevant audio warnings
  4. optional conversation subject last
- The optional subject should preserve the quiet look from the mock: medium weight, small text, muted white, left-aligned with the status block.
- Generic subjects such as "wants to talk" should stay hidden because they repeat what the surface already says.

## What changed this session

- Runtime mock subject text was removed from `ContentView`; the app no longer injects `"Dinner tonight"` in debug builds.
- `TurboCallPrototypeView` still supports an optional `requestSubject` for the future backend-backed subject.
- The subject is now below telemetry, matching the preferred order above.
- Local volume warnings are treated as telemetry/action context, not as the subject.

## What is not working

- Real request subjects are not yet wired from backend/app state into the call screen.
- The subject component is preserved only as an optional view path until the real request subject exists.

## Recommended next step

1. Wire real request subject data into the app/backend contract when the request-subject feature is implemented.
2. Pass the real optional subject into `TurboCallPrototypeView`.
3. Keep the order `name -> status -> telemetry -> subject` unless a future design pass deliberately changes the hierarchy.
