# Handoff 2026-06-04 18:07

## Summary

Fast Relay foreground testing between @mau iPhone and @bau iPad repeatedly reached a state where audio stopped and the receiver stayed blocked as if the sender was still talking. The latest evidence points to a receiver-side receive-turn/liveness failure: stale media and stale backend transmit projection could keep the receive side alive after the turn should have closed, and one UI-facing pending-playback check could block behind the audio playback lock.

## Current truth

- Latest physical evidence came from `/tmp/turbo-debug/20260604T154914Z-postfix-stuck-sender-repro`.
- In that run, @mau sent `transmit-195`; @bau accepted the first Fast Relay packet and sent first playback ACK, then emitted almost no diagnostics for about 25 seconds while the UI appeared frozen.
- The silence timeout that should have cleared the receive turn did not run on time. When it finally ran, backend still projected `transmit-195` as active even though its lease had expired.
- A stale Fast Relay packet after the repair could still reopen playback before this session's fix.
- Direct QUIC foreground back-and-forth looked healthy in development. The failing shape is currently Fast Relay receive turnover after repeated turns.
- This is classified as app/backend contract plus app media adapter/effect executor, not a pure Apple/PTT/audio activation bug.

## What changed this session

- `PCMWebSocketMediaSession.hasPendingPlayback()` no longer blocks waiting for `receivePlaybackLock`; if the lock is busy, it logs a budgeted diagnostic and returns `true` so MainActor timeout/repair logic does not freeze behind audio scheduling.
- Added DEBUG hooks to lock/unlock the receive playback lock for a focused nonblocking proof.
- `handleIncomingAudioPayload` now rejects incoming audio before playback when the selected backend transmit lease is already expired, logs the drop, and triggers local expired-lease repair.
- Added/used focused Swift tests for:
  - pending-playback snapshot not blocking behind the receive playback lock
  - stale Fast Relay packet after expired backend transmit lease being dropped before playback
  - relay continuity/queue/sender-clock stale packet receive-turn stop behavior
  - expired backend transmit lease timeout/watchdog repair
  - direct QUIC incoming audio handler not blocking the next MainActor packet handler

## What is not working

- The physical Fast Relay repeated-turn cell has not yet been retested after this fix.
- Hypothesis still to validate on device: diagnostics/logging load may make the playback lock stall easier to hit, but the app should now fail closed rather than freezing the receiver UI or reviving stale playback.
- The current ACK design is still too weak. "First playback ACK" proves only that one packet was accepted/scheduled; it does not prove the current receive epoch is healthy, current, draining, or still owned by the active transmit.

## Recommended next step

1. Launch the new build on @mau and @bau, force Fast Relay, and repeat foreground count-to-10 turns until either it stays healthy or the invariant fires again.
2. On failure, pull diagnostics immediately and search for `Receive playback lock busy while checking pending playback`, `Dropped incoming audio after expired backend transmit lease`, `transmit.live_projection_after_lease_expiry`, `media.incoming_audio_queue_delay`, and receiver stuck UI timing.
3. If the physical cell still fails, promote the new evidence into one receive-epoch reducer/projection test before touching physical devices again.
4. Next architecture pass: replace first-playback ACK with an algebraic receive-turn ACK contract:
   - identity: `channelID`, `transmitID`, sender device, receiver device, transport lane, sequence floor, started time, lease expiry
   - monotonic receiver state: `notSeen -> accepted -> playbackScheduled/started -> acked` or `rejected(reason)`
   - ACK/negative-ACK carries epoch identity, sequence/frame evidence, source, and accepted/rejected reason
   - sender matches ACKs only against the current epoch; stale ACKs and stale packets cannot revive old turns
   - timeout and repair live in one reducer/owner, with websocket/relay/audio as adapters
