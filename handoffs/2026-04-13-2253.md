## What is not working

- Real locked-screen wake is still not fully reliable on physical devices.
- The latest narrowed failure shape is:
  1. sender enters wake path
  2. bridge sends wake push successfully
  3. receiver logs `Incoming PTT push received`
  4. receiver reconnects websocket and buffers wake audio
  5. receiver never logs `PTT audio session activated`
- That means the remaining failure is at the Apple PushToTalk activation boundary, not the backend wake-target path.
- The local app fallback path is intentionally deferred until foreground while locked, so a missing `PTT audio session activated` currently means the receiver will wait rather than silently start app-managed lock-screen playback.
