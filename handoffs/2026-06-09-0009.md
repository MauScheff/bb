## What changed this session

- Added a foreground-banner epoch to `IncomingBeepSurfaceState`.
- Beeps created before the current foreground-banner epoch are marked surfaced and cannot become top banners.
- Starting foreground activation clears active/pending foreground banner surfaces and begins a new epoch.
- `handleForegroundBeepNotification` now classifies delivery using:
  - `currentApplicationState()`
  - `protectedDataAvailableProvider()`
  - foreground activation suppression
- Inactive/locked notification callbacks are consumed as background-delivered receipts and do not queue pending foreground surfaces.
- Activation suppression remains active through the first backend poll so backend catch-up cannot replay an old Beep.
- Added targeted Swift tests for the lock/inactive path, epoch suppression, epoch-eligible foreground Beeps, background app-open replay, and true foreground banner behavior.

## Notes

- The important design distinction is arrival surface vs durable Beep state.
- The in-app top banner is an arrival surface only.
- Backend convergence is still useful for real foreground state convergence, but only for Beeps in the current active/unlocked foreground epoch.
- Apple delivered-notification enumeration is not sufficient as the sole replay guard because lock/unlock timing can remove or transform notification evidence before app catch-up runs.
