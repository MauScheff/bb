## Work Item 1: Route Matrix Proof

Problem statement:

The app moved from a simple home-selected contact flow to a detail/focused contact flow. The old path was fast because selection established intent and started prewarm early. Every route must preserve that property.

Routes to prove:

- `@mau` enters `@bau` detail, taps Ask, `@bau` accepts.
- `@bau` enters `@mau` detail, taps Ask, `@mau` accepts.
- Sender taps Ask from detail, then backs out to home before receiver accepts.
- Receiver accepts from home.
- Receiver accepts while already viewing a contact/detail surface.
- Tapping an active call row from home bypasses detail and opens the call screen.

Likely owner:

- App/client projection and navigation state.
- Backend truth only if diagnostics show the backend relationship/channel state is wrong.

Narrowest proof lane:

- Start with Swift reducer/projection tests around selected-contact intent, focused contact, and selected-peer state.
- Add simulator scenarios only for route transitions that need UI/navigation journey evidence.

Recommended first step:

1. Search for existing tests around `detailFocusSelectionRunsSelectedContactPrewarmPipeline`, notification accept, and `selectedPeerProjectionReadDoesNotMutateRequestedSelectionState`.
2. Add a small route-matrix test table that asserts each route establishes the same selected contact and prewarm evidence before join.

Suggested invariant framing:

- `selected.intent_missing_before_join`: join/accept path started without selected-contact intent for the contact when the route had an explicit contact source.

## Work Item 2: UI State Invariants

Problem statement:

During physical testing we saw impossible or confusing UI states:

- call screen showed the peer as merely “online”
- requested/detail/call state flapped
- sender sat in call screen connecting while receiver only saw request
- accepting a request did not immediately show the call screen
- end/minimize left stale detail stack state

Likely owner:

- App/client projection for one-device contradictions.
- Merged diagnostics for pair contradictions.
- Backend only if backend relationship/channel truth contradicts itself.

Narrowest proof lane:

- App-side invariant checks in projection/diagnostics for local contradictions.
- Merged diagnostics rules for pair/convergence contradictions.
- Swift tests for projection rules that should be locally impossible.

Candidate invariant IDs:

- `selected.call_visible_without_join_evidence`: call screen visible for incoming request before local accept/join evidence exists.
- `selected.call_visible_peer_online`: call screen visible while selected peer projection is plain online/ready-to-request rather than connecting/connected.
- `selected.request_call_flap`: selected route alternates requested and call-visible states without a new user/backend event that justifies the route change.
- `selected.stale_detail_after_call_end`: detail/focused route remains active after explicit call end/minimize should return to home.

Recommended first step:

1. Inspect existing `AppDiagnostics` invariant checks before adding new IDs.
2. Prefer adding one high-value invariant first: call screen must not show plain online/requestable peer state.
3. Prove it with one Swift projection test.

## Work Item 5: Prewarm Contract

Problem statement:

The fast path depends on prewarm starting when contact intent is known. Regressions happened when detail UI delayed or skipped that selected-contact intent, causing Ask/Accept to pay serial setup costs.

What the fast path warms:

- local media shell
- selected-peer prewarm hint over websocket
- Direct QUIC offer/probe or remote offer request
- foreground talk path readiness after local join
- relay prejoin when enabled
- foreground notification prewarm for incoming requester

Likely owner:

- App/client selected-contact coordination and prewarm pipeline.

Narrowest proof lane:

- Swift tests around prewarm event emission and idempotence.
- No timing-based correctness tests. Use event/order assertions and diagnostics facts; treat latency as telemetry only.

Required properties:

- entering detail starts selected-contact prewarm
- foreground notification starts incoming-contact prewarm
- repeated prewarm for the same contact is idempotent
- Ask/Accept does not wait for prewarm to complete
- backend truth still wins over local optimistic/prewarm evidence

Recommended first step:

1. Strengthen `detailFocusSelectionRunsSelectedContactPrewarmPipeline`.
2. Add an explicit idempotence test for repeated detail focus and repeated notification prewarm.
3. Keep timing out of the invariant; only measure accept-to-ready as performance telemetry.

Suggested invariant framing:

- `selected.prewarm_missing_for_explicit_intent`: explicit contact intent did not enqueue the selected-contact prewarm pipeline.
