# Handoff 2026-04-13 20:15

## Summary

Receiver audio readiness is now backend-authoritative instead of an ephemeral client-side websocket hint. The backend persists current-session receiver readiness, exposes it through `/v1/channels/:channelId/readiness/:deviceId` as `audioReadiness`, and uses it to gate transmit-target resolution. Swift now derives hold-to-talk gating from that canonical readiness snapshot, the local and hosted route probes are green, the non-UI Swift suite is green, and real-device foreground PTT/audio is working.

## Current truth

- Backend `/readiness` is the source of truth for distributed receiver readiness.
- The authoritative wire contract now includes nested `audioReadiness`:
  - `self.kind`
  - `peer.kind`
  - `peerTargetDeviceId`
- Swift selected-peer derivation reads backend `audioReadiness` instead of maintaining a separate parallel remote-readiness store.
- Backend transmit-target resolution now requires the chosen receiver device to be current-session audio-ready.

- Exposed that state through `/readiness.audioReadiness` and enforced it in backend transmit-target resolution.
- Collapsed the Swift side to derive remote audio readiness from `TurboChannelReadinessResponse` instead of a parallel side map.
