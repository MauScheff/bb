## What is not working

- Permission prompts can appear at the worst possible time. In this run, the Local Network prompt appeared during the direct QUIC connection attempt and caused the app to resign active, which closed the media runtime and canceled the QUIC attempt. This specific prompt has now been accepted on `@mau`, so it should not block the immediate next probe.
- Follow-up direct QUIC `answer`, `ice-candidate`, and `hangup` signals from the interrupted run were later ignored by `@bau` because the `@bau` attempt had already been reset after media closed. This was downstream of the Local Network permission interruption, not the original receiver-gate bug.
- Backend still advertises `supportsDirectQuicUpgrade=false`, so Force Probe currently depends on the debug bypass path until the backend capability is enabled.
- Backend stale-presence invariants are still present in merged diagnostics:
  - `presence.stale_active_channel_on_session_connect`
  - `channel.stale_peer_presence_projected_live`
  - `channel.stale_self_presence_projected_live`
  These are not the active direct QUIC permission blocker, but they remain a backend correctness issue.
