## What changed this session

- Production Direct QUIC identity slice was staged earlier in the work:
  - added a production identity manager that creates/reuses a per-device local P-256 key and matching self-signed certificate for Network.framework TLS identity use;
  - added diagnostics for provisioning status, active fingerprint, and identity source;
  - extended backend/device route and probe surfaces for Direct QUIC identity metadata;
  - added Direct QUIC authorization checks to use backend-authoritative peer certificate fingerprint metadata;
  - kept global Direct QUIC upgrade capability gated separately from provisioning support.
- Debug `.p12` identity remains only as a fallback/diagnostic path.
- Direct QUIC production payloads should not set `debugBypass`.
- Request-state self-healing was added after the user saw both apps show Connect and then a request from one side did not appear on the other:
  - `BackendSyncEvent.invitesPartiallyUpdated(incoming:outgoing:now:)` was added.
  - `BackendSync.applyPartialInvites(incoming:outgoing:now:)` was added.
  - `refreshInvites()` now fetches incoming and outgoing independently instead of failing the whole invite sync if either side fails.
  - Partial invite success is applied and the failed side's prior state is preserved.
  - Foreground talk-request notifications now refresh contact summaries and invites, then verify that the request projected locally.
  - If a foreground request notification still does not project as an incoming request, the app records `request.foreground_notification_not_projected` and schedules one short recovery retry.
- The request-state fix is meant to make the system recover on both sides when backend truth exists but local projection misses it.

## What is not working

- Foreground Direct QUIC audio cold path is still not reliable on physical devices.
- The user believes this worked before the certificate work. That is a useful clue but not proof that certificates caused the regression.
- Strong hypothesis:
  - the production identity/certificate work changed enough Direct QUIC connection or authorization timing to expose an existing transmit/startup race;
  - or the new identity path causes the app to take a different Direct QUIC/control-plane path than the old debug certificate path;
  - or a recent warm-path optimization changed the handoff ordering between backend lease, receiver prepare, system audio activation, Direct QUIC media readiness, and sender release.
- Observed failure sounds more like an activation/ordering bug than pure certificate verification:
  - receiver only starts showing remote talking after sender release;
  - warm path can work later;
  - diagnostics include activation-lifetime invariant failure.
- The odd request/invite state should be retested first with the latest build before continuing audio debugging.
- There is a risk that optimized audio packet logging now logs only first-packet events, so diagnostics may be less chatty than older captures. Do not interpret fewer packet logs as no packets without checking the newer first-packet/summary fields.
