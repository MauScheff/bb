## Summary

Direct QUIC manual device testing now works for foreground media: both devices reached Direct, `@mau` transmitted over `transport=direct-quic`, and `@bau` received Direct QUIC audio payloads and scheduled playback. ## Current truth

- Both physical devices have ready debug Direct QUIC identities imported through the in-app PKCS#12 diagnostics path.
- Manual Force Probe established Direct QUIC media on both devices.
- Diagnostics proved actual media delivery over Direct QUIC: the receiver logged repeated `Direct QUIC audio payload received` events and playback buffer scheduling.
- The apparent hearing failure was likely local output routing/headphones, not media transport failure.
- Background/lock currently tears down or loses the live QUIC path as expected; the required behavior is automatic re-probe/re-promotion after wake or path loss.
- The backend still advertises `supportsDirectQuicUpgrade=false`, so DEBUG automatic probes intentionally reuse the same debug bypass as the Force Probe button.

## What changed this session

- Added automatic Direct QUIC probe wrapper and scheduler in `PTTViewModel+DirectQuic.swift`.
- Path loss now falls back to relay immediately and schedules an automatic Direct QUIC reprobe after retry backoff.
- Media session startup now calls the automatic probe wrapper instead of the manual-only probe path.
- App foreground activation now attempts an automatic probe for the selected contact.
- Direct activation cancels any pending automatic reprobe task.
- Media runtime now tracks and cancels `directQuicAutoProbeTask` alongside the existing promotion timeout/probe controller lifecycle.
- Verification passed with `just swift-test-target directQuicUpgradeRuntimePreservesDirectPathForLateCandidateUpdates`: 565 tests passed.

## What is not working

- Physical background/lock validation is not yet done on the new auto-reprobe build.
- When the foreground device is the dialer-answerer role and the listener-offerer is suspended, it may not be able to initiate a new offer until the listener wakes. This is inherent in the current deterministic role split.
- The `ptt: Failed to clear active remote participant before local transmit` diagnostic still appears on device sometimes. It is not blocking Direct media now, but it should be tracked because it means PushToTalk participant state can lag app/media state.
- Production identity provisioning is not implemented. `.p12` import is debug-only.

## Notes

- Production should not ship `.p12` identity import as the provisioning model. Each device should generate its own non-exportable private key locally, register the public certificate/fingerprint with the backend under that device identity, and use backend-signed device metadata to validate Direct QUIC peer fingerprints during offer/answer.
- The production identity should be per device, not per handle. Handles/accounts own device registrations; device registrations own Direct QUIC key material and revocation/rotation state.
