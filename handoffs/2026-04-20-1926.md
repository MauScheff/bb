# Handoff 2026-04-20 19:26

## Summary

This handoff captures the current Swift-side post-refactor state of the Apple PTT debugging loop. The main refactor slices are in place and local proof is green again, but there is still a real device regression in the `foreground -> background` wake path: when the receiver locks, the sender can fall to `Establishing connection... / Waiting for peer` instead of showing wake UI.

## Current truth

- The large Swift refactor is materially in place:
  `ConversationDomain` now derives explicit durable session, execution, and control-plane projections.
  `SelectedPeerSession` stores the typed projection instead of only flattened UI state.
  readiness publication, receive execution, and transmit task orchestration have explicit coordinator/reducer seams.
- Local proof is green:
  targeted Swift runs passed and the full Swift suite passed with `359 tests`.
- The current blocker is device-side `fg -> bg`, not basic `fg <-> fg`.
- Latest merged diagnostics show the receiver background transition is not simply “peer disappeared.”
  The sender receives the live `app-background-media-closed` signal, briefly knows the peer is `wakeCapable`, and then a lagging backend refresh overwrites that with a stale waiting/connected mix.
- The most recent patch is in the channel-readiness merge path, not APNs or audio transport.

## What changed this session

- Investigated the latest merged diagnostics around the background transition.
- Confirmed the failure shape:
  the foreground sender receives fresh wake metadata after the peer backgrounds, but the next backend refresh drops the sender back to `Establishing connection...` because backend peer connectivity lags stale-`true`.
- Tightened the wake-capable merge in [`PTTViewModel+BackendSync.swift`](/Users/mau/Development/Turbo/Turbo/PTTViewModel+BackendSync.swift):
  preserve live `wakeCapable` metadata across a transient background refresh when the previous session was genuinely routable, while still rejecting stale wake metadata during initial handshake cases.
- Added regression coverage in [`TurboTests.swift`](/Users/mau/Development/Turbo/TurboTests/TurboTests.swift):
  `fetchedWaitingReadinessDoesNotPreserveWakeCapableWhenPeerDeviceIsConnectedAndBackendIsStillReady`
  `fetchedWaitingReadinessPreservesWakeCapableWhenPeerConnectivityLagsDuringBackgroundTransition`
  `fetchedWaitingReadinessDoesNotPreserveStaleWakeCapableWhenExistingSessionWasNotRoutable`

## What is not working

- Current device blocker:
  after one peer locks, the other can still regress to `Preparing audio...` -> `Establishing connection...` -> `target user has no connected receiving device in this channel` with a disabled `Waiting for peer` button, instead of staying in wake-capable state.
- This is a regression from earlier behavior and from earlier parts of this same debugging arc.
- Hypothesis, now narrower than before:
  if the latest merge fix does not fully solve it on device, then there is at least one more overwrite path besides `refreshChannelState(for:)` that is still allowed to replace live wake metadata with lagging backend state.
  The likely remaining candidates are selected-session sync or another backend refresh path, not APNs token loss.
