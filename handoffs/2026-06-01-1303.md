- The current public edge is intentionally temporary:
  - nginx owns TCP `443`
  - relay is off
  - UDP `443` is not yet owned by the new VM stack
- The current shape does not prove the final target of one Turbo-owned edge for HTTP/WebSocket, TCP fallback relay, and QUIC.
- A Google Application Load Balancer is not the right center of gravity if Turbo wants app-owned QUIC. It can support HTTP/3 client-to-load-balancer, but it terminates that protocol at Google's edge rather than giving the Rust service end-to-end control of custom QUIC streams/datagrams.

## Turbo Edge Plan

### Purpose

`turbo-edge` is the single public front door for Turbo.

Target external shape:

```text
staging.beepbeep.to:443
  TCP -> turbo-edge
  UDP -> turbo-edge
```

Target internal shape:

```text
turbo-edge
  -> runtime over localhost/internal HTTP or a typed Rust boundary
  -> relay modules for QUIC and TCP fallback media
  -> Postgres/Redis only through runtime-owned effect paths unless a measured edge-local cache is introduced
  -> compiled Unison kernel only through runtime-owned semantic commands
```

### Responsibilities

- Own public TCP `443`.
- Own public UDP `443`.
- Terminate TLS for HTTPS/WebSocket compatibility.
- Proxy or dispatch current HTTP routes to `turbo-runtime`.
- Proxy or dispatch current WebSocket routes to `turbo-runtime`.
- Host QUIC media relay over UDP `443`.
- Later host QUIC control streams over UDP `443`.
- Host TCP fallback relay over TCP `443` if TCP fallback remains required.
- Emit one edge diagnostic stream for:
  - TLS accept
  - HTTP route
  - WebSocket upgrade
  - QUIC handshake
  - QUIC stream/datagram
  - TCP fallback relay
  - runtime dispatch latency
  - relay forwarding latency

### Non-Responsibilities

- Do not make `turbo-edge` the durable source of Conversation, Participant, Device, or Talk Turn truth.
- Do not let Redis become durable truth.
- Do not call Unison per packet, heartbeat, websocket ping, or QUIC datagram.
- Do not introduce a passthrough load balancer before the single-VM edge is proven.

### Implementation Shape

Start small and keep the existing deployed runtime intact.

Recommended repo shape:

```text
runtime/
  src/bin/turbo-edge.rs        # first binary, if keeping edge close to runtime
  src/edge.rs                  # TCP HTTPS/WebSocket proxy and public config
  src/edge_diagnostics.rs      # edge events and timing

relay/
  src/transport_quic.rs        # reuse for UDP media path
  src/transport_tcp.rs         # reuse for TCP fallback path
  src/protocol.rs              # reuse relay frames
  src/relay_state.rs           # keep in-memory relay state for single VM stage
```

Current Cargo reality:

- Workspace [`Cargo.toml`](/Users/mau/Development/Turbo/Cargo.toml) has `members = ["runtime"]`.
- `relay` is excluded from the workspace but `runtime` already depends on it by path.
- The VM Dockerfile currently builds `turbo-runtime` and `relay` separately.

First slice:

```text
turbo-edge TCP 8443 locally
  -> terminate TLS with local/dev certs
  -> reverse proxy all HTTP/WebSocket traffic to runtime 8091
  -> prove health route and websocket upgrade behavior
```

Second slice:

```text
turbo-edge TCP 443 on VM
  -> replace nginx
  -> use /etc/letsencrypt/live/staging.beepbeep.to certs
  -> preserve current HTTPS health behavior
```

Third slice:

```text
turbo-edge UDP 443 on VM
  -> host existing relay QUIC media path
  -> keep runtime control over Talk Turn and relay grants
  -> prove physical-device direct/relay media smoke
```

Fourth slice:

```text
turbo-edge QUIC control
  -> add control streams after the HTTP/WebSocket path is stable
  -> keep HTTP/WebSocket as compatibility and debug lane until mobile support is proven
```

### Passthrough Network Load Balancer Rule

Do not add the passthrough Network Load Balancer now.

Use direct VM static IP while:

- there is one staging VM
- there are no users
- fast deploy/debug speed matters more than failover
- `turbo-edge` is still being shaped

Add a passthrough Network Load Balancer when one of these becomes true:

- more than one edge VM is needed
- VM replacement should not require DNS changes
- health-checked failover is needed
- production ingress should be separated from instance lifecycle

Expected later shape:

```text
staging.beepbeep.to
  -> passthrough Network Load Balancer IP
  -> one or more turbo-edge VMs
       TCP 443
       UDP 443
```

Use passthrough, not Application Load Balancer, for the unified Turbo edge because the backend must receive and terminate the real TCP/UDP traffic.
