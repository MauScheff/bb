## Summary

Investigated whether Turbo telemetry should live in Unison Cloud, Cloudflare Web Analytics, Cloudflare Log Explorer, or Cloudflare Workers Analytics Engine. The recommendation is to use a new Cloudflare telemetry Worker that writes structured domain events into Workers Analytics Engine, modeled after the existing APNs worker pattern. Keep Unison as the control-plane authority and optional source for exact support bundles or canonical incident records, but do not turn the Unison control-plane database into the main telemetry firehose.

## Current truth

- Turbo already has a working Cloudflare Worker integration pattern for APNs in [`cloudflare/apns-worker/`](/Users/mau/Development/Turbo/cloudflare/apns-worker), and Unison already syncs Worker config into the production environment via [`turbo_cloudflare_apns_worker_integration.u`](/Users/mau/Development/Turbo/turbo_cloudflare_apns_worker_integration.u).
- The APNs worker is deliberately a transport adapter only: Turbo remains authoritative for wake-target selection, begin-transmit rules, and wake diagnostics. That is the right model to reuse for telemetry.
- The current backend/control-plane architecture and docs push Unison storage toward query-shaped `OrderedTable` designs, small transactions, and authoritative product state, not a large general-purpose analytics/event warehouse. See [`BACKEND.md`](/Users/mau/Development/Turbo/BACKEND.md) and [`Server/backend_architecture.md`](/Users/mau/Development/Turbo/Server/backend_architecture.md).

## What changed this session

- Compared Unison Cloud storage against Cloudflare telemetry products with the specific Turbo use case in mind: bug reconstruction, operational diagnosis, and product-usage analytics.
- Verified that the APNs worker pattern is reusable:
  - shared-secret auth from backend to Worker
  - narrow Worker responsibility
  - Wrangler-managed deploy
  - Unison-managed env/config sync into production
- Identified the recommended split:
  - Workers Analytics Engine for the main event stream and aggregate queries
  - Discord for high-signal alerts only
  - optional Unison and/or R2 for exact deep incident artifacts later
- Identified the main implementation constraint for client telemetry:
  - do not embed a long-lived Worker secret in the iOS app
  - either proxy client events through backend/Unison first, or mint short-lived signed ingest tokens for the client

## What is not working

- No telemetry Worker exists yet.
- No normalized event schema exists yet for app and backend to emit the same event envelope.
- No query or dashboard layer exists yet for reconstructing sessions, pair timelines, failure-rate spikes, or usage funnels.
- Hypothesis: Workers Analytics Engine should be the primary telemetry store, while Unison remains optional for exact incident artifacts. ## Recommended next step

1. Create `cloudflare/telemetry-worker/` by copying the deployment/secret/config shape of [`cloudflare/apns-worker/`](/Users/mau/Development/Turbo/cloudflare/apns-worker), but with routes like `GET /health` and `POST /telemetry/events`.
2. Add Unison integration next to the APNs integration so `turbo.deploy` can sync:
   - `TURBO_TELEMETRY_WORKER_BASE_URL`
   - `TURBO_TELEMETRY_WORKER_SECRET`
   - optional `TURBO_TELEMETRY_DISCORD_WEBHOOK`
3. Define one shared event envelope before any app work:
   - identity: `eventName`, `timestamp`, `source`, `sessionId`, `deviceId`, `userHandle`, `peerHandle`
   - build/runtime: `appVersion`, `iosVersion`, `backendVersion`
   - diagnosis: `severity`, `invariantId`, `phase`, `reason`, `reportId`
   - metrics: durations or counters in doubles
   - compact metadata blob for structured details
4. Start with backend-originated telemetry only:
   - join/leave/reconnect/restore/teardown decisions
   - background/foreground presence transitions
   - invariant violations
   - wake delivery outcomes
   - selected-peer/backend readiness transitions
5. Query those events through the Analytics Engine SQL API first, and only add dashboards/Discord alerts after the event vocabulary is stable.
6. Add client telemetry second, with one of these auth models:
   - preferred first cut: client sends high-value events to backend, backend forwards to Worker
   - more advanced cut: backend mints short-lived signed ingest tokens for direct client-to-Worker writes
7. Keep raw, verbose transcripts out of Analytics Engine. If we later need deep bundles, store them separately in Unison or R2 and link them from the event stream by `reportId`.

## Notes

- The strongest argument against using Unison as the main telemetry database is not capability, it is workload shape. Unison is well suited to authoritative, query-shaped, typed state and exact incident records. Analytics-style event firehoses are a different operational problem.
- The strongest argument against using Log Explorer is not price, it is product fit. It appears to be centered on Cloudflare-native logs and datasets rather than custom mobile/session event ingestion. ŅłĆciteŅłéturn1view3Ņłéturn1view4Ņłéturn1view5Ņłü
- The strongest argument for Analytics Engine is that it matches the actual problem:
  - Worker-native ingestion
  - structured events
  - SQL queries
  - high-cardinality dimensions
  - cheap aggregate analytics
  - simple path to Discord alerting
- Recommended first dashboards/queries:
  - session join success rate by build
  - unexpected leave rate by build and iOS version
  - first-audio latency by matrix leg
  - background-presence failures by device model
  - top invariant IDs by app version
