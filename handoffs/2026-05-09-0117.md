# Handoff 2026-05-09 01:17

## Summary

This handoff captures the call-screen stability work after device testing exposed two user-visible issues: hold-to-talk could remain latched after lifecycle/proximity interruptions, and established calls could visually bounce from Connected to Connecting when the WebSocket briefly reconnected. The intended product behavior is now: keep real diagnostics faithful, but smooth harmless control-plane churn in the call UI once a call is already established.

## Current truth

- The repo was clean when this handoff was written: `git status --short` produced no output.
- The call screen now treats `Establishing connection...` as user-facing `Connecting...`.
- Established calls get a bounded 10 second control-plane reconnect grace. This grace only starts when there is real local session evidence: selected contact, joined local PTT session, active channel match, prior connected session continuity, and matching system PTT session.
- During that grace, stale WebSocket reconnect artifacts such as downgraded remote audio readiness do not make the call UI fall back to Connecting or disable Talk.
- Hold-to-talk now force-releases on lifecycle interruptions such as app resign-active/background. This covers SwiftUI `DragGesture` cancellation cases where `.onEnded` may not fire.
- The call screen peer telemetry line now scopes connection information to the peer and only adds `Direct` when the media path is direct. Relay is intentionally not shown as a label in that line.

## What changed this session

- Added lifecycle transmit cancellation through `cancelActiveTransmitForLifecycleInterruption(reason:)`.
- Changed `endTransmit` to accept a reason and to clear/sync the local press latch immediately.
- Added `liveCallControlPlaneReconnectGraceStartedAt` and `liveCallControlPlaneReconnectGraceSeconds`.
- Added WebSocket idle/connecting hooks that start reconnect grace, and connected hooks that clear it.
- Made reconnect grace a first-class selected-session / conversation-derivation input instead of only overloading relay readiness.
- Collapsed user-facing `Establishing connection...` status text to `Connecting...`.
- Updated call context UI to render peer connection as:
  - `Mellow’s connection · Wi-Fi`
  - `Mellow’s connection · Wi-Fi · Direct`
  - `Mellow’s connection · Direct` if no peer bearer telemetry is available but direct transport is known.
- Added regressions around lifecycle transmit cancellation and established-call WebSocket reconnect smoothing.

## Notes

- The current design does not use two WebSockets as a stability strategy. The preferred pattern remains one logical resumable control connection, idempotent backend commands, replay after reconnect, and UI grace for harmless transients.
- The reconnect grace should not hide real disconnection forever. It is intentionally bounded and evidence-gated.
