# Handoff 2026-05-11 15:00

## Summary

Physical-device call setup improved from roughly 14-15 seconds with UI flicker to roughly 9 seconds with both devices converging to ready. The next architectural task is to introduce a single typed control-command/event envelope and one ingestion entry point for backend HTTP, websocket, Direct QUIC, relay, PTT callbacks, timers, and user intents.

## Current truth

- Backend remains authoritative for distributed truth: channel membership, readiness, request state, wake targeting, and active transmit authority.
- Direct QUIC and websocket lanes are valid fast paths for non-authoritative hints, peer-local control, media-path preparation, and speculative UI continuity.
- Fast-path events must merge only when they match the current channel/session/attempt epoch; stale events must become no-ops.
- Latest physical-device observation: both @mau and @bau reached ready in about 9 seconds.
- Latest merged diagnostics showed Direct QUIC auto-probe was skipped because relay-only mode was forced: `blockReason=relay-only-forced`.
- The previous @bau call-screen drop was fixed by preserving accepted backend joins during hosted visibility lag instead of clearing pending local join after 8 seconds.
- Swift test bundle passed after the latest client changes: 872 tests in 4 suites.

## What changed this session

- Incoming request handling now treats accepted/handled requests as consumed by request epoch, preventing stale request resurfacing when older backend projections arrive.
- Backend join settling now lasts 20 seconds, so a successful backend join can remain locally connecting while hosted membership/readiness projection catches up.
- Regression coverage was added for accepted join visibility lag and stale incoming request suppression.
- Successful backend join responses are now projected immediately into local backend sync as self-membership proof, so the app no longer has to wait for the next hosted channel-state refresh before clearing the local side of the join.
- Regression coverage was added for the accepted-join projection: `acceptedBackendJoinProjectionMarksLocalMembershipBeforeHostedRefresh`.
- Diagnostics confirmed the remaining setup delay is not the old flicker bug; it is now mostly backend/control-plane visibility plus duplicated readiness publication/refresh churn.

## What is not working

- Call setup is still too slow at about 9 seconds.
- Direct QUIC is not contributing to setup in the latest run because diagnostics mode still has relay-only forced.
- Control commands do not yet all share one envelope or one intake machine. Current command/event handling is split across:
  - `BackendCommandCoordinator`
  - `BackendSyncCoordinator`
  - `ControlPlaneCoordinator`
  - `SelectedPeerReducer`
  - direct Direct QUIC handlers
  - websocket signal handlers
- Readiness signals are repeatedly published/received after ready, creating noisy logs and likely unnecessary refresh work.

## Recommended next step

1. Add a design doc or implementation scaffold for a unified `ControlEventEnvelope` and `ControlEventIngestor`.
2. Define the algebraic merge rules before moving behavior:
   - duplicate event or irrelevant epoch is identity/no-op
   - commands are idempotent by `operationID` or `eventID`
   - independent hints commute when they touch disjoint projections
   - backend authoritative snapshots dominate conflicting hints
   - local/direct hints may improve UI only inside a bounded current epoch
3. Start with a narrow integration: route websocket signal envelopes and Direct QUIC receiver prewarm/ack/path-closing through the new intake, without changing backend authority.
4. Then use the same intake for optimistic join/readiness hints to reduce setup latency while preserving backend correction.
5. For the immediate 9-second path, first disable relay-only in diagnostics and re-run the foreground connect test; then inspect whether Direct QUIC prewarm/control hints arrive before backend readiness.

## Notes

- Desired envelope fields: source, conversation/channel ID, contact ID, local/remote device IDs, session epoch, attempt ID, operation ID, event ID, timestamp, payload, and authority level.
- Suggested authority levels: authoritative backend snapshot, authoritative local PTT callback, peer hint, transport hint, timer, user intent.
- The single-entry-point goal is primarily correctness and observability. Latency improvements should come from using the same entry point to accept fast current-epoch hints while waiting for slower backend confirmation.
