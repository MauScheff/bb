# Handoff 2026-05-13 11:13

## Summary

The app is back at a fast, stable physical-device baseline after the selected-contact/detail-screen freeze regression. This handoff is specifically for the missing work: restoring the UX/proof coverage we temporarily backed away from, without losing the 2-3 second accept-to-ready path that selected-contact prewarm gives us.

## Current truth

- `main` is at `0e5c722` (`checkpoint`) at the time this handoff was written.
- The latest user-verified baseline is good:
  - tapping a contact no longer freezes the app
  - Ask projects immediately and starts the local cooldown without blocking the UI
  - accepting a request can reach ready on both physical devices in about 2-3 seconds
- The freeze class was client-side:
  - SwiftUI projection/read helpers were mutating selected-peer coordinator state during render
  - selected-peer projection reads must remain pure
  - synchronization belongs in explicit tasks, events, reducers, command handlers, or lifecycle hooks
- Selected-contact prewarm is the main known fast-path win. The important rule is: contact intent must be known before Ask/Accept, not discovered for the first time inside the join path.
- Detail/contact prototype is enabled again, with selected-contact intent moved to contact detail focus.
- Automatic diagnostics publish is debug default-on for physical-device iteration, with `TURBO_IOS_AUTOMATIC_DIAGNOSTICS_PUBLISH=0` as the escape hatch.
- Higher-volume reducer/state capture diagnostics stay opt-in unless a targeted run needs them:
  - `TURBO_IOS_REDUCER_TRANSITION_DIAGNOSTICS=1`
  - `TURBO_IOS_STATE_CAPTURE_TELEMETRY=1`
- Fast-path mechanics and rationale are documented in [`journal/2026-05-13-0108.md`](/Users/mau/Development/Turbo/journal/2026-05-13-0108.md).

## What is not working

- No active physical-device blocker is known at this handoff.
- Missing work is mostly proof, route coverage, and careful UI restoration.
- The detail flow is not yet proven across all route combinations:
  - Ask from `@mau` detail, accept on `@bau`
  - Ask from `@bau` detail, accept on `@mau`
  - Ask from detail, back to home before accept
  - accept from home versus accept while already inside a contact/detail route
  - active call tap from home should bypass detail and go directly to the call screen
- UI-state invariants are still mostly conceptual. They need executable coverage:
  - call screen must never present the peer as merely “online”
  - requested/detail/call state must not flap between requested and call screen
  - minimizing or ending a call should return to home, not leave stale detail navigation state
  - accepting a request should show the call screen immediately even if the user is on detail
- Merge diagnostics still mixes historical and current violations. This makes clean runs harder to interpret after a bad run.
- Readiness latency is still mainly evaluated by physical-device observation. Avoid treating a hard number as correctness; use phase/order invariants for correctness and latency telemetry for performance.

## Recommended next step

1. Add route-matrix proof for detail/home/notification parity at the smallest useful layer first. Start with Swift reducer/projection tests for selected-contact intent and navigation state, then add simulator scenarios only where the journey adds evidence.
2. Add executable UI-state invariants for the impossible screens listed above. Register any new invariant IDs in [`invariants/registry.json`](/Users/mau/Development/Turbo/invariants/registry.json) if they need durable diagnostic identity.
3. Split merge diagnostics into active/current violations versus historical violations so a repaired run is visibly clean.
4. Continue UI polish only after the route matrix remains stable:
   - keep the current call screen largely unchanged
   - refine home contact rows with larger lightweight rows, avatar/initials, spacing instead of hard dividers
   - keep detail top icons as native material floating controls
   - keep detail Ask as the solid primary action

## Notes

- The most important performance lesson is that selected-contact prewarm removes serial work from accept/join. It starts or coalesces routing discovery, peer warm hints, Direct QUIC offer/probe work, media shell setup, receiver readiness, and relay/prejoin work before the user accepts.
- The most important correctness lesson is that view reads must stay pure. A SwiftUI body/projection read cannot “helpfully” repair selected-session state.
- The local optimistic Ask state is intentionally not backend truth. It is pending-command evidence with a bounded lifetime; backend state still wins.
- Automatic diagnostics publish is intentionally debug default-on again because physical-device iteration benefits from fresh diagnostics. If it causes responsiveness issues, turn it off per run with `TURBO_IOS_AUTOMATIC_DIAGNOSTICS_PUBLISH=0` and inspect publish cadence before changing app logic.
- If the next run regresses, use the dev-debug low-token pattern first and save raw diagnostics to `/tmp/turbo-debug` before inspecting narrow excerpts.
