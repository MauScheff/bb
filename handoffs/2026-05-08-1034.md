## What changed this session

- Backend `/v1/config` now includes a `directQuicPolicy.stunServers` list.
- `scripts/direct_quic_provisioning_probe.py` now verifies:
  - `directQuicPolicy` is an object;
  - when `supportsDirectQuicUpgrade=true`, `stunServers` is a non-empty list;
  - each STUN server has a non-empty `host`;
  - each `port`, when present, is a valid integer UDP/TCP port.
- Production was deployed with `just deploy-force` after normal deploy showed service-name/custom-domain propagation flapping between old and new config.
- Swift Direct QUIC probe startup now attempts a one-shot local production identity repair and device re-registration when the identity is missing/failed, instead of immediately and repeatedly skipping the probe.
- Swift now preserves the original Direct QUIC production identity provisioning failure metadata and flushes `ios.direct_quic.identity_provisioning_failed` after backend telemetry is available. The event includes the label, device id, localized Keychain/certificate error, identity status/source, fingerprint, provisioning status, and flush reason.
- Swift outgoing audio now fails closed when receiver-readiness gating times out. It records `Timed out waiting for remote receiver audio readiness; not sending outbound audio` and throws `remote receiver audio was not ready`, so the app surfaces media failure instead of sending relay packets into a receiver that cannot play them.

## What is not working

- We currently depend on Google public STUN. That is acceptable as a temporary unblock, but it creates an external infrastructure dependency and exposes source IP/timing metadata for Direct QUIC candidate discovery to Google.
- Preferred next phase: use Cloudflare Realtime managed STUN/TURN so Direct QUIC candidate discovery and relay fallback can run on Cloudflare's edge network without operating our own UDP service.
- Cloudflare Workers are not a good fit for a custom STUN server today because Workers are HTTP/WebSocket oriented and expose outbound TCP sockets, not inbound arbitrary UDP datagrams.
- Cloudflare Spectrum can proxy UDP at Layer 4, but it is a proxy to an origin, not a place to run packet-handling application code. It may require an Enterprise plan and still needs an origin STUN service.
- Cloudflare Realtime offers managed STUN/TURN at `stun.cloudflare.com` / `turn.cloudflare.com`. It still requires backend-generated short-lived TURN credentials and client TURN candidate support.
- Unison Cloud does not support UDP, so it cannot host STUN directly.
- TURN is not implemented in the client. STUN-only improves direct-path success, but relay fallback remains the production safety net for symmetric NATs or UDP-hostile networks.
- TURN should not be treated as a small extension to a Worker STUN plan. TURN requires an actual UDP relay surface, allocation state, credentials, permission checks, bandwidth limits, and abuse controls. It can run beside STUN in the same origin service, but it is a materially larger product/infra commitment.
- Backend presence/session repair is still suspect. The backend already emits invariants when stale connected-session truth survives an offline transition, but production showed the repair can fail under current storage/session behavior. That needs a backend-owned follow-up, likely around `turbo.service.presence.offline`, `turbo.store.sessions.disconnectCurrent`, and readiness projection after background/offline transitions.

## Recommended next step

1. Create/configure Cloudflare Realtime TURN credentials in Cloudflare.
2. Add backend config/env for the Cloudflare TURN key id/API token, following `UNISON.md` env sync rules.
3. Add a backend route or config surface that returns short-lived ICE server policy for the authenticated device/session.
4. Update the iOS Direct QUIC policy model to parse STUN and TURN URLs/credentials, not only `{ host, port }` STUN servers.
5. Extend the client candidate gatherer to request TURN relay candidates from Cloudflare and include them in the existing Direct QUIC signaling payloads.
6. Keep WebSocket relay fallback unchanged; do not block transmission on STUN/TURN success.
7. Add probes:
   - `/v1/config` or dedicated ICE policy probe proves managed STUN/TURN policy is present when Direct QUIC upgrade is enabled.
   - a TURN credential probe proves backend-issued credentials are short-lived and accepted by Cloudflare.
   - a physical-device Direct QUIC test proves relay candidates can nominate when direct STUN candidates cannot.
8. Later, revisit self-hosted `coturn` or custom Rust only if we need a BeepBeep-owned hostname/service, cost control, custom policy, or reduced managed-provider dependency.

## Notes

- STUN is intentionally simple: parse a Binding Request, preserve the transaction ID, return a Binding Success Response with `XOR-MAPPED-ADDRESS`.
- A custom Rust implementation remains reasonable for STUN-only if we later decide to own that service and keep the scope narrow:
  - UDP socket on `0.0.0.0:3478`;
  - RFC 5389 Binding Request/Success only;
  - no authentication;
  - no relay allocation;
  - no long-term credentials;
  - no TURN permissions/channels;
  - packet-size and rate limits;
  - structured request counters and sampled logs.
- `coturn` remains the safer self-hosted production path if the goal becomes protocol/hostname ownership. A Rust service is attractive for STUN-only if we want a tiny audited component and are comfortable owning abuse/rate-limit behavior.
- The cleanest Cloudflare-native managed option is Realtime TURN/STUN. Spectrum can sit in front of a UDP origin if we want to run `coturn` ourselves, but a Worker alone should not be planned as the STUN implementation surface unless Cloudflare ships inbound UDP Workers.
- A custom Rust TURN implementation should be considered a separate project, not a quick add-on to STUN.
