## What is not working

- Main unresolved product seam:
  the app can hold a stale local websocket/control-plane belief after resets or degraded connectivity, while the backend has already lost the socket/session. That can surface as:
  - `Backend connection failed` briefly on home screen after reset/network transition
  - `backend invite sync failed`
  - `contact sync failed`
  - sessions that look locally connected while backend runtime says no live socket exists
- Important nuance:
  this is not currently a hosted invite-route regression. The routes are up and semantic invite flows work on hosted.
- Hypothesis:
  the next source fix is probably app-side bootstrap/control-plane retry or stale-socket reconciliation, not another backend route composition patch.
- Secondary open item:
  `Backend connection failed` after network reset should likely be treated as bootstrap-recoverable instead of user-visible fatal-looking status when the app subsequently reconnects on its own.

## Recommended next step

1. Add a client-side bootstrap/control-plane recovery path so transient startup failures after network reset auto-retry instead of surfacing as a terminal-looking home-screen error.
2. Add a foreground/session-drift recovery rule for the case where local state still thinks websocket/session is alive but backend refresh/runtime indicates the socket is gone.
3. Update `scripts/route_probe.py` so the `/join` semantic check first establishes the websocket/session precondition, then rerun `just route-probe` to make the probe fully green again.
