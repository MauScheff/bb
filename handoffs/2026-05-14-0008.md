- Physical-device stability issues reported earlier in the thread are not closed by this handoff:
  - later disconnect/reconnect fluctuations
  - occasional fallback/transition oddities around call state changes
- Hosted canary timing is not yet consistently green, so performance still needs another clean pass after the design work settles.
- Hypothesis:
  the remaining stale-state risk is no longer primarily "presence rows exist too long", but "some routes may still treat non-authoritative runtime facts as current without a shared epoch/lease check".

## Recommended next step

1. Audit every backend runtime fact used in call-critical routes and classify it as either:
   durable monotonic fact, or leased/epoched runtime fact.
2. Extend the current lease model so all call-critical projections derive from typed current-state predicates rather than raw rows:
   presence, readiness, signaling authorization, relay/transport session, active transmit target.
3. Add invariants at the authoritative seams for each of those predicates, then rerun hosted probes and physical-device tests only after the backend model is tightened.

## Notes

- The key design rule for the next session is:
  stale rows may exist, but stale rows must never be sufficient to project current truth.
- The right mathematical split here is:
  durable state should converge monotonically, while runtime state should be lease/epoch/fencing based.
- Do not treat secondary indexes as truth. They are candidate-finders only.
