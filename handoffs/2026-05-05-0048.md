# Handoff 2026-05-05 00:48

## Summary

Foreground, Direct QUIC, and foreground-to-background PushToTalk receive are now working well again after the latency and wake-receive regressions. The important latency win was moving backend APNs wake delivery off the sender's `begin-transmit` critical path, while preserving a synchronous `scheduled` wake diagnostic. The important correctness win was making wake activation monotonic: a late incoming PTT push can confirm/enrich an already activated receive flow, but it can no longer downgrade `systemActivated` back to `awaitingSystemActivation`.

## Current truth

- The backend deployed at `https://beepbeep.to` now schedules wake push asynchronously from `turbo.service.channels.internal.sendWakePushUnlessTargetAudioReady`.
- `begin-transmit` no longer waits for APNs delivery. It records a `scheduled` dev wake event, forks the actual APNs send, and returns the lease response.
- Synthetic proof after deploy showed the `scheduled` wake event recorded roughly 75 ms after `startedAt`. The later APNs result was recorded separately on the worker path.
- Repeated synthetic `begin-transmit` calls after warm-up were around 0.7-0.8 seconds instead of the 3-4 second regression.
- Device testing after the client wake fix is reported by the user as working well.
- The current best latency shape is: local/system PTT handoff + backend lease + already-warmed media path. It no longer includes APNs send time, Direct QUIC promotion time, or receiver media setup time when the session is warmed.

## What changed this session

- Backend:
  - `turbo.service.channels.internal.sendWakePushUnlessTargetAudioReady` now appends a `scheduled` wake event synchronously when the target is not already audio-ready.
  - The actual `sendWakePushIfPossible` call now runs inside `forkAt pool()` instead of blocking `begin-transmit`.
  - If the target is already audio-ready, the backend still records `skipped-target-ready` and emits the debug diagnostic asynchronously.
  - Deployed with `just deploy`.
- Client:
  - Fixed `WakeReceiveState.confirmingIncomingPush(...)` so confirmation preserves the current activation mode instead of always returning `.awaitingSystemActivation`.
  - This prevents the failure where Direct QUIC audio arrived on the background receiver but was buffered behind an activation that had already happened.
  - Added `wakeExecutionReducerConfirmIncomingPushDoesNotDowngradeSystemActivation()`.
- Proof:
  - The full Xcode test run built and executed 605 tests; the new regression passed in that run.
  - The full suite still has unrelated existing failures, mostly simulator/local-backend tests around `127.0.0.1:9` and wake-grace expectations.

## How we made first press optimal

- We stopped treating APNs delivery as part of the sender's startup path. APNs is necessary to wake a background receiver, but the sender should not wait for Apple's push response before starting local transmit.
- We kept observability by writing a synchronous `scheduled` wake event before forking APNs. This preserves merged diagnostics without putting APNs latency back on the user path.
- We moved receiver readiness earlier:
  - foreground media prewarm after the session becomes ready
  - automatic Direct QUIC promotion and warm ping once a peer session is connectable
  - Direct QUIC receiver prewarm request before transmit when a direct path exists
  - backend/control-plane prepare signals before the first audio chunk when available
- We kept the no-missing-audio invariant by gating first outbound audio briefly when the peer is wake-capable but receiver audio readiness has not recovered yet.
- We made wake activation monotonic. Once Apple has granted a system PTT audio session for receive, later control-plane confirmation cannot move that flow backwards into "waiting for activation."
- Net effect: first press should now spend time only on the parts we cannot fully pre-complete: system PTT handoff/activation, backend lease, and any cold process/network wake. It should not pay for Direct QUIC setup, APNs response, or receiver media creation when those were already warmed.

## Notes

- The prior regression came from making APNs wake send synchronous in order to prove wake delivery. That proved delivery but put APNs latency back into the sender path. The correct version keeps the proof signal (`scheduled`) synchronous and the APNs delivery asynchronous.
- The background receiver failure was not a Direct QUIC transport failure. Diagnostics showed Direct QUIC audio was received repeatedly, but the receiver had been downgraded to `awaitingSystemActivation` and buffered audio instead of playing it.
- For latency, preserve the no-missing-audio invariant over speed. A one-second press that never drops audio is better than a half-second press that can lose the first word.
