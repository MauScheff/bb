# Journal 2026-06-01 11:34

## Summary

The self-hosted Unison-kernel/Rust-runtime lane reached release-readiness evidence. The new backend path keeps the existing Cloud backend intact while proving a parallel architecture where Unison owns pure Conversation, Participant, Device, Connection, and Talk Turn decisions, and Rust owns HTTP, WebSocket, QUIC/TCP relay, APNs, Postgres, Redis, metrics, and effect execution. The final blocking proof was physical-device lock-screen APNs wake, which passed after recopying iPad diagnostics and rebuilding the manifest around the real wake window.

## Problem

- The original architecture plan needed implementation-grade proof, not only documentation.
- The self-hosted path needed parity and rollback evidence before any production cutover could be considered.
- Physical-device proof still lacked one Apple/PTT/audio cell: lock-screen APNs wake from iPhone to iPad.
- The first lock-screen collection sent APNs successfully but failed to copy receiver diagnostics from the iPad because CoreDevice closed the file-transfer socket.

## Design formulation

- Rust owns effects. Unison owns meaning.
- The Unison kernel boundary is pure: command plus state snapshot plus policy snapshot plus kernel version returns a decision, effect plan, invariant events, and replay facts.
- The Rust runtime interprets effect plans through durable storage, live routing, APNs, sockets, relay transport, diagnostics, and metrics.
- The current Unison Cloud backend remains a rollback and comparison lane until a separate cutover decision.
- Physical Apple/PTT/audio behavior stays a physical-device proof cell; lower lanes prove the shared state and runtime rules, but lock-screen wake requires real devices.

## What changed

- Added the self-hosted Rust runtime and proof harnesses for HTTP, WebSocket, Postgres/Redis integration, scenario fuzzing, shadow comparison, rollback probing, and readiness aggregation.
- Extracted the Rust relay into reusable protocol, state, auth, metrics, QUIC, and TCP modules under the broader runtime lane.
- Added kernel/runtime terminology, proof architecture, command tables, physical-device gates, readiness gates, and rollback rules to `UNISON_KERNEL_RUST_RUNTIME.md`.
- Added TLA/model-check coverage for the Talk Turn actor protocol.
- Added physical-device boundary collection, manifest, proof, merge, finalize, and status tooling.
- Added APNs print/send helpers and device diagnostics launch/copy improvements.
- Hardened Swift app/runtime diagnostics and tests around talk turns, wake readiness, transport selection, audio send fallback, and physical-device evidence.
- Added `target/` to `.gitignore` so root Cargo build output is not committed.

## What worked

- The readiness gate aggregates release evidence instead of relying on a single smoke test.
- Shadow comparison kept Cloud and self-hosted behavior visible at the semantic boundary.
- Physical proof cells were collected independently and finalized into one canonical proof artifact, so a failed collection did not invalidate passing cells from other runs.
- Recopying receiver diagnostics after the CoreDevice transfer failure preserved the real lock-screen wake evidence.
- The lock-screen wake cell passed with receiver-side evidence: background/locked state, system wake activation, PTT audio activation, audio chunks, and scheduled playback.

## What not to repeat

- Do not treat APNs status `200` as lock-screen wake proof by itself. Receiver-side audio and playback evidence are required.
- Do not use a collector timestamp as the evidence window when the actual device event began earlier. The manifest must cover the real wake window.
- Do not use physical devices as the first lane for kernel/runtime semantics. Use them only for Apple/PTT/audio/hardware boundaries.
- Do not cut over production just because readiness is green. Green readiness authorizes a staged deployment and canary, not a blind switch.
- Do not commit root `target/` build output.
