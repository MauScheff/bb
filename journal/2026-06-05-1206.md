## Problem

- Previous symptom: after repeated fast back-and-forth turns, a sender could build stale outbound audio and the receiver could process packets many seconds late, producing missing turns, receiver tails, and `media.outbound_audio_transport_backpressure_drop`.
- Fixed context: physical foreground-to-foreground, forced fast relay, `@mau` iPhone and `@bau` iPad, debug build `1.0 (170)`.
- New symptom to investigate next: on the last iPad attempt, holding the talk button briefly moved into the transient talking/listening startup state, then fell back to present/idle instead of holding a live transmit.
- Concrete new diagnostic window on `@bau`: `2026-06-05T10:03:11.172Z` begin transmit, `10:03:12.051Z` system transmit began, `10:03:12.751Z` startup completed, `10:03:12.754Z` PTT audio session deactivated, `10:03:12.758Z` treated as system transmit end while `coordinatorPressing=true` and `transmitPressActive=true`.

## Design formulation

- Stale outbound audio is an epoch law: once a transmit epoch is no longer current, queued and in-flight audio for that epoch must be cancelled, not delivered and not reported as transport success.
- Transport cancellation is not a media transport failure. It is the normal result of a newer epoch or explicit stop superseding older work.
- The iPad bounce is a separate Apple/PTT/audio boundary problem until proven otherwise: local transmit should not be cancelled merely because a PTT audio deactivation arrives while the user is still pressing and the system transmit has just begun.
- Candidate invariant for next work: `ptt.audio_deactivation_during_active_local_press`. If `coordinatorPressing == true`, `transmitPressActive == true`, and the current PTT audio owner is `local-transmit:<id>`, a deactivation must either be proven to be an Apple/system-ended event for that same transmit or converted into a bounded reactivation/retry path, not immediately collapse the user-visible state without cause.

## What changed

- `AudioChunkSender` now treats `CancellationError` from the send hot path as an epoch stop: it clears pending payloads, cancels in-flight send tasks, and does not report transport failure.
- `PTTViewModel+TransmitAudioSend` now checks the outgoing audio target token around async boundaries in direct and fast-relay send paths, throwing cancellation when the token is stale.
- Added a focused Swift regression proving cancelled transport sends clear the sender without reporting failure.
- The prior stale relay receive reset fix remains important: old relay deliveries after receive reset are dropped instead of being admitted into a new receive epoch.

## What worked

- Physical retest after the stale-epoch cancellation fix: repeated foreground fast-relay counting turns worked well again, especially with normal human spacing between turns.
- Stress cadence still matters, but the catastrophic stale-tail/no-audio pattern appears substantially improved.
- The user no longer sees the prior spontaneous app-minimize behavior in this test path.
- The latest bad behavior is now narrower and cleaner: not stale outbound backlog, but a PTT/audio activation-deactivation race on local press startup.

## What not to repeat

- Do not treat a stale audio send closure returning without sending as success. That creates false delivery and lets old audio survive across turns.
- Do not debug every missing-audio report as playout/jitter first. If logs show stale outbound drops or delayed receive-queue admission, classify epoch/backpressure ownership before touching playout.
- Do not paper over the iPad bounce by leaving the UI in talking state when Apple/PTT has already deactivated. The visible fallback is safer than pretending audio is live. The fix should explain or prevent the deactivation.
- Do not let app-managed audio prewarm/route refresh fight Apple-gated PTT transmit ownership. In the failed window, route-change refresh and capture restarts happened around the deactivation and should be audited.
