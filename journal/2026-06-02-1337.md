# Journal 2026-06-02 13:37

## Summary

The self-hosted Rust runtime needed a stronger Talk Turn monotonicity fence: a released conversation could be granted again with the same epoch, which made stale releases and downstream notifications ambiguous. I turned that into a production-shaped regression in the fuzz wrapper and preserved the failure as a structured oracle instead of a traceback.

## Problem

- A second request after release reused `talkTurnEpoch = 1` instead of advancing.
- The production-shaped scenario showed the mismatch only after the runtime move, under real Postgres/Redis and the live HTTP process.
- The concrete failure became visible as a stale release after a newer grant and a reused epoch in the follow-on request.

## Design formulation

- The invariant is monotonic Talk Turn epoch progression per conversation, even across release/regrant cycles.
- The state-machine rule is that stale releases must fail closed and not disturb a newer grant.
- The right shape is backend-owned, because the runtime is the authority for committed Talk Turn state and post-commit effects.

## What changed

- Extended `backend/scripts/self_hosted_production_scenario_fuzz.py` to check duplicate renew/release, a newer grant after release, and a stale release against the newer grant.
- Added socket-level `OSError` handling so connection resets are captured as structured fuzz evidence instead of a Python traceback.

## What worked

- `just self-hosted-scenario-fuzz-local 123 3 ...` passed with the new stale-release oracle.
- `just self-hosted-http-probe ...` passed after the probe was separated onto a distinct Conversation fixture.
- `just beepbeep-backend-gate` passed after the runtime/probe behavior and the production-shaped oracle were aligned.
- The follow-up `just reliability-fuzz-self-hosted-overnight 424242 100 ...` stayed green.

## What not to repeat

- Don’t let a process-level connection reset hide the actual failing response.
- Don’t use the same active Conversation for compatibility smoke and stale-release regression checks when the goal is to test independent routes.
