# Journal 2026-06-07 18:25

## Summary

Direct QUIC and Fast Relay packet media now keep true transport arrival time into playout and can carry single `VoicePacketV1` Opus packets as raw datagram bytes instead of JSON/base64 strings. This narrows the early-turn hitch by removing two artificial costs: delayed local handler time being interpreted as network jitter, and avoidable string framing on byte-capable lanes.

## Problem

- Physical tests over Direct QUIC and Fast Relay had good control-plane convergence but audible startup gaps, usually around the first one or two seconds of a turn.
- Diagnostics repeatedly showed `media.incoming_audio_queue_delay` / `Incoming audio payload was delayed in the local receive queue`.
- The important contradiction was that packets could already be local, but SwiftNetEq saw them as late because the receive executor handed them to playout after local scheduling work.

## Design formulation

- Transport ingress time is a packet fact. Playout must use the time the transport received the packet, not the later time when a local actor/executor happens to process it.
- SwiftNetEq is the jitter cushion for packet media. A second legacy scheduler startup cushion on the same packet stream is double buffering.
- Direct QUIC and Fast Relay are byte-capable lanes. When the app payload is already a single binary `VoicePacketV1`, the lane should carry those bytes directly and keep string/base64 encoding as compatibility fallback.

## What changed

- Added a transport-arrival-aware media-session path so `LiveAudioReceiveExecutor` passes `IncomingAudioIngressContext.receivedAtNanoseconds` into playback.
- `PCMWebSocketMediaSession` now preserves `transportReceivedAtNanoseconds` through its pending remote audio queue and passes that timestamp into `SwiftNetEqPlayoutEngine.insert`.
- SwiftNetEq packet playout now reports `.alreadyCushioned`, avoiding an extra legacy startup cushion.
- Added Direct QUIC binary datagram framing for packet audio with `TQD1` magic.
- Added Fast Relay binary datagram framing for packet audio with `TRD1` magic.
- Updated the Rust media relay to parse Fast Relay binary packet datagrams, validate sender/session ownership, and forward the original datagram bytes unchanged.
- Added focused Swift and Rust codec/timing proofs.

## What worked

- Targeted Swift proofs confirm that live receive playback gets the transport arrival timestamp, steady arrivals stay at base delay, late inter-arrival gaps still raise target delay, and Direct/Fast binary packet datagrams round-trip.
- Rust relay protocol and relay tests confirm binary datagrams decode while JSON datagram join compatibility remains intact.
- This addresses a source-level timing mistake rather than hiding the symptom with more buffering.

## What not to repeat

- Do not tune SwiftNetEq using local handler delay as if it were network jitter.
- Do not add another receive-side startup cushion for packet media when SwiftNetEq is already doing the cushion.
- Do not judge Direct QUIC or Fast Relay live quality while wrapping packet bytes in JSON/base64 if the lane can carry raw datagrams.
