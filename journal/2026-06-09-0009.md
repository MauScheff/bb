# Journal 2026-06-09 00:09

## Summary

Foreground Beep banners now distinguish true foreground arrival from app activation after background or lock-screen delivery. The product rule is: Apple/system notifications own background and lock-screen delivery; the in-app top banner is only for Beeps that arrive during the current active, unlocked foreground epoch. Backend Beep projection still converges contact/detail state, but it must not replay a top banner for a Beep the user was already shown by the system.

## Problem

- A Beep sent while the receiving device was backgrounded or locked could show the Apple notification and then replay the in-app foreground banner when the app became visible.
- The first fix consumed delivered notifications on launch/app-active, but physical testing still reproduced the banner when the app was already underneath the lock screen and the user only unlocked the device.
- The concrete missing case was: `@bau` app open -> lock screen -> `@mau` sends Beep -> Apple lock-screen notification appears -> unlock without tapping notification -> app immediately shows the foreground banner.

## Design formulation

- Foreground banners are an arrival affordance, not the durable Beep state.
- Backend Beep projection remains useful for real foreground state convergence: if the app is active and the Apple foreground callback is late or absent, backend state may still surface the banner.
- That convergence is only valid inside the current active, unlocked foreground epoch.
- A Beep whose `sentAt` / `createdAt` precedes the current foreground-banner epoch is treated as already delivered outside the foreground surface and is marked seen-without-banner.
- Notification callbacks received while inactive, locked, or during foreground activation are classified as background-delivered receipts. They update Beep/contact state, but they do not queue pending foreground surfaces.

## What changed

- Added foreground-banner epoch state to `IncomingBeepSurfaceState`.
- Added reducer event `foregroundBannerEpochStarted(Date)`.
- Suppressed any candidate whose sent time is before the current foreground-banner epoch.
- Cleared active and pending foreground surfaces when a new epoch starts.
- Added `protectedDataAvailableProvider` so notification handling can classify locked/inactive delivery in tests.
- Changed `handleForegroundBeepNotification`:
  - true active + protected data available + no activation suppression => foreground banner path
  - inactive/locked/activation-suppressed => consume as background-delivered without pending banner
- Kept foreground activation suppression active through the first backend poll, not just delivered-notification intake.
- Added regressions for inactive/locked notification delivery, pre-epoch suppression, post-epoch banner eligibility, background open suppression, and true foreground notification banner behavior.

## What worked

- Physical device test confirmed the lock/unlock case no longer replays the in-app foreground banner.
- True foreground Beep notifications still show the in-app top banner.
- Backend convergence remains available for Beeps created after the active foreground epoch.
- The fix is model-level: the reducer encodes banner eligibility, instead of relying only on Apple delivered-notification enumeration, which can miss lock/unlock timing.

## What not to repeat

- Do not treat `willPresent` or backend projection as automatically foreground-eligible without checking lifecycle and lock state.
- Do not rely only on `UNUserNotificationCenter.getDeliveredNotifications()` to suppress replay. It is useful evidence, but lock/unlock can remove or transform delivered notifications before app catch-up runs.
- Do not hardcode `applicationState: .active` in notification reconciliation paths. Use `currentApplicationState()` and the foreground-banner epoch.
- Do not remove backend convergence entirely; it is needed for real foreground arrivals where backend state beats the transient Apple callback.
