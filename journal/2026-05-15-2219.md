# Journal 2026-05-15 22:19

## Summary

Physical-device reliability testing between `@lau` and `@bau` exposed a cluster of distributed-state and audio-projection bugs after the recent refactor. We hardened the selected-peer/talk state machine so backend settling, transmit prepare signals, playback drain, explicit leaves, and relay path changes no longer project impossible UI states. We also made route probes tolerate hosted Unison read-after-write visibility lag, deployed the backend, and reached a clean automated Swift/probe proof set; the remaining proof is the next fresh physical-device smoke.

## Problem

- The app could flap between `ready`, `connecting`, `talking/listening`, relay, and fast relay even when the user-visible call should have been stable.
- The receiver could keep showing talking/listening after audio had ended, blocking the talk button for several seconds.
- A transmit prepare signal could make the receiver look like the peer was talking before real audio arrived.
- Pressing End on one device could leave the other side in a call screen or connecting state after the local side had returned to contacts.
- Foreground talk-request notifications could race or duplicate in a way that made the first Connect tap appear inert.
- Deployment probes assumed immediate invite/channel projection visibility, but hosted Unison sometimes exposed a pending invite before all derived reads saw it.

## Design formulation

- The selected-peer UI must be a projection of typed control-plane and audio facts, not a loose bundle of latches.
- `ready` with recoverable backend drift is still user-operable when the transport/audio path is live; only real playback drain should block counter-talk.
- A `ptt-prepare` signal is a wake/readiness hint, not remote speech. It must not project the peer as talking without a transmit start and audio path evidence.
- Explicit local leave is a fenced user intent. It should not be cleared until backend membership has converged enough that the opposite device cannot resurrect the call screen.
- Talk request handling should become convergent and idempotent at the model/action level: duplicate surfaces and stale invite IDs must resolve to the same current pending request when the pair/channel identity matches.
- Hosted route probes should test user-observable convergence, not instantaneous storage/projection visibility after a write.

## What changed

- Added a stable `readyHoldToTalkDisabled` selected-peer detail for the narrow case where the call is ready but playback drain must temporarily block local transmit.
- Kept hold-to-talk enabled through backend settling and remote transmit-stop grace when the selected path is otherwise live.
- Stopped treating `ptt-prepare` as remote talking; it now arms wake/readiness without creating a false listening/talking projection.
- Preserved explicit leave latches until backend membership convergence, including background/system leave callbacks.
- Reasserted wake join readiness after reconnect so stale receiver readiness does not destabilize the next transmit.
- Added call telemetry and tests around transmit begin/end, playback drain, backend readiness drift, and leave convergence.
- Updated `scripts/route_probe.py` with `run_eventually_check` so invite and channel-state checks poll boundedly for hosted projection convergence.
- Deployed the backend after Swift and route-probe proofs passed.

## What worked

- The fixes targeted the ownership boundary that produced each visible symptom: client projection for selected-peer state, audio coordinator classification for prepare-vs-speech, backend/probe convergence for hosted read-after-write behavior, and leave fencing for call teardown.
- Focused Swift tests reproduced the important bad projections without requiring repeated device runs.
- The full Swift suite passed after the changes.
- Route probes passed against `https://beepbeep.to` after eventual checks were added.
- `just deploy` succeeded and the postdeploy synthetic conversation passed functionally.

## What not to repeat

- Do not treat every transient backend mismatch as `connecting`; that hides the real distinction between recoverable settling and a dead path.
- Do not use the physical devices as the main iteration loop for shared projection bugs once a reducer/domain test can express the invariant.
- Do not project remote speech from control-plane preparation alone. Speech requires transmit/audio evidence.
- Do not make route probes depend on single-read immediacy when the production platform can expose bounded eventual visibility.
- Do not solve notification races only at the banner surface; the correct fix is a convergent request model plus one idempotent accept/connect action.
