# Journal 2026-05-06 01:55

## Summary

First-talk startup now treats Direct QUIC as a bounded preference, not a hard requirement. A newly connected Direct QUIC-capable session may spend up to 5 seconds in a Direct QUIC first-talk grace window; if Direct QUIC activates, transmit starts on the fast path, and if it does not, the UI falls back to relay-backed `Connected` instead of staying stuck in `Connecting...`.

## Problem

- Fresh connections sometimes became fast only when Direct QUIC happened to be warm, but otherwise we waited several seconds on Apple/system activation or showed odd transient states.
- Requiring Direct QUIC before declaring the peer fully connected was too strict because not every connection has a viable direct path.
- Allowing relay immediately lost the chance to reproduce the observed fast first hold-to-talk path where Direct QUIC was available within roughly the first few seconds.
- The concrete design pressure was: "give Direct QUIC a few tries, but allow connected without QUIC if it cannot come up."

## Design formulation

- Ownership boundary: selected-peer readiness owns whether the user can hold to talk; Direct QUIC owns path promotion; diagnostics observe rather than mutate startup state.
- Invariant: `Connected` must not require Direct QUIC forever. Direct QUIC may block the first-talk projection only during a bounded grace window when Direct QUIC is advertised or already attempting.
- State-machine rule: `firstTalkStartupProfile == .directQuicWarming` projects to `waitingForPeer(.localTransportWarmup)` and blocks transmit; `.relayWarm` projects to ready when relay/media/backend readiness are otherwise satisfied.
- This shape keeps the fast path available without turning an optimization into a liveness dependency.

## What changed

- Added `FirstTalkStartupProfile` as a typed projection: `directQuicWarm`, `directQuicWarming`, `relayWarm`, `relayWarming`, `unavailable`.
- Added a per-contact, per-channel Direct QUIC first-talk grace record in media runtime.
- Added a 5 second grace expiration task that refreshes selected-peer state and captures diagnostics when relay fallback becomes allowed.
- Limited grace eligibility to sessions where backend advertises Direct QUIC or an explicit Direct QUIC attempt already exists.
- Added diagnostics field `firstTalkStartupProfile`, with a non-mutating read path so pulling diagnostics does not start the grace timer.
- Added regression coverage for the projection waiting during Direct QUIC grace and falling back to relay after expiry.

## What worked

- The key fix was making Direct QUIC a startup profile in the selected-peer projection rather than scattered checks in button handling.
- Existing relay/WebSocket tests initially caught an over-broad DEBUG behavior: debug Direct QUIC bypass was making plain relay sessions enter the grace window. Tightening grace eligibility fixed that without weakening the new invariant.
- User device verification after the change reported that the flow felt better. There is still a short wait on some first attempts, but it is acceptable for now because liveness is preserved and subsequent warm presses remain fast.

## What not to repeat

- Do not make diagnostics calls start timing windows or trigger warmup side effects. Diagnostics should report the current projection, not create it.
- Do not let DEBUG Direct QUIC bypass affect ordinary relay readiness. Debug bypass is useful for explicit probe work, but first-talk grace should follow actual backend capability or an existing attempt.
- Do not represent this with booleans like `isDirectWarm` and `isRelayWarm`; the mutually exclusive startup modes need a sum type so projection logic stays auditable.
