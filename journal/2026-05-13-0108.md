# Journal 2026-05-13 01:08

## Summary

Restoring selected-contact prewarm made the physical-device accept-to-ready path fast again: the user observed roughly 2-3 seconds from accepting a request to both devices ready. The important lesson is that the old home-screen flow was fast because contact intent was known before the join path, not because the home screen itself was special.

## Problem

- After introducing the contact detail flow, request/accept often took much longer than the earlier 3-6 second path.
- Some runs also froze the app when tapping a requested contact.
- The freeze came from a UI projection read mutating selected-peer coordinator state during SwiftUI rendering.
- Once that render-time mutation was fixed, prewarm had been left disabled as a stabilizing step, which made the flow correct but slower.

## Design formulation

- Selecting or focusing a contact is real user intent and should start best-effort preparation before ask/accept.
- Reading UI projection state must be pure. A view read may derive display state, but it must not synchronize coordinators, enqueue effects, publish diagnostics, or mutate selected-session state.
- The detail screen must preserve the same selected-contact intent contract as the old home screen: entering detail for a contact should make that contact the active prewarm target.
- Notification-driven accept should also establish the same contact intent before join work, because the user has implicitly selected the requester.

## What changed

- `selectedPeerProjection(for:)` / `selectedPeerState(for:)` are protected by a focused Swift test so projection reads do not mutate requested selected-peer state.
- Selected-contact prewarm is restored to debug default-on, with `TURBO_IOS_SELECTED_CONTACT_PREWARM=0` as the escape hatch.
- Reducer transition diagnostics and automatic diagnostics publishing remain opt-in so proof tooling does not overload the app during physical testing.
- The detail/contact-action prototype was re-enabled after the selected-contact intent path was restored for focused contact entry.
- Outgoing ask now projects a bounded optimistic outgoing request immediately. The backend write still owns durable truth, but the UI no longer waits for `createInvite` to return before showing the requested state and 30-second cooldown.

## What worked

- With selected-contact prewarm restored and the render-time mutation removed, physical device testing returned to the desired fast path.
- User-observed result: accepting a request reached ready on both devices in about 2-3 seconds.
- This supports the hypothesis that prewarm before the authoritative join path is the main 80/20 latency win.

## What The Fast Path Warms

The selected-contact pipeline is not a single optimization. It bundles several small, idempotent preparations that remove serial work from the accept/join path:

- `media-shell`: creates a local media session shell for the focused contact while the app is foregrounded, not transmitting, and not already inside a PTT audio session. This avoids paying media object setup at the moment the call needs to become interactive.
- `peer-prewarm-hint`: sends a websocket `selected-peer-prewarm` control signal to the peer using fresh backend routing. On the receiver, this can create the receiver-side media shell or, if that contact is also selected, run the receiver's selected-contact prewarm pipeline.
- `direct-quic-prewarm`: starts the Direct QUIC offer/probe path or asks the peer to offer when this device is not the listener-offerer. This can make the direct path available before first talk instead of discovering it after accept.
- `foreground-talk-prewarm`: after a local joined session exists, resumes the websocket if needed, optionally starts app-managed playback-only local media, publishes local receiver audio readiness, starts automatic Direct QUIC probing, and requests receiver prewarm for first talk.
- `media-relay-prejoin`: when relay debug/override is enabled and a peer target device is known, connects relay receive transport early.
- `foreground notification prewarm`: incoming foreground request notifications run a related path even without list selection: media shell, peer hint, and Direct QUIC prewarm for the requester.

This explains the 2-3 second result: accept no longer has to do routing discovery, local media shell creation, peer warmup signaling, Direct QUIC negotiation, receiver-readiness publication, and transport preparation as one long chain. Much of that work has already started, coalesced, or completed by the time the user accepts.

## Ask Tap Responsiveness

Pressing Ask is an asynchronous command. The UI should immediately reflect local user intent while the backend invite write is in flight:

- project the contact as `outgoingRequest`
- start the 30-second "Ask again" cooldown locally
- replace the optimistic projection when the backend invite is seeded
- clear the optimistic projection if the backend command fails

This is intentionally not a backend patch. It is local pending-command evidence with a bounded lifetime, so stale optimistic state can expire and authoritative backend state still wins once it arrives.

## Prewarm Scope

Not every stage should be widened to every contact.

- Safe global or all-contact candidates: keep foreground websocket routing healthy; keep backend contact summaries/presence fresh; keep local identity/provisioning ready; keep low-cost routing metadata fresh.
- Safe selected/detail candidates: media shell creation, peer prewarm hint, Direct QUIC probe/offer, receiver prewarm request, and relay prejoin. These imply stronger user intent and should stay tied to selected/focused contact, detail entry, foreground notification, or another explicit intent source.
- After detail returns, entering the detail screen should immediately establish selected-contact intent and run the selected-contact prewarm pipeline. Pressing the detail primary action should not be the first time the system learns which contact is likely to be used.
- If we later support multiple high-probability contacts, model that explicitly as a bounded "warm candidates" set with priorities and resource limits rather than silently prewarming every heavy subsystem for every contact.

## What not to repeat

- Do not move the UI to a detail screen in a way that delays contact intent until the user presses the final action.
- Do not let projection helpers "helpfully" sync selected peer state from inside view reads.
- Do not compensate for missing contact intent with arbitrary join timeouts; the faster and cleaner fix is to start the idempotent prewarm pipeline at the moment intent becomes known.

## Next

1. When reintroducing the detail screen, enter-detail must call the same selected-contact/prewarm intent path that home-screen selection uses.
2. Add an automated scenario or lower-level reducer proof that detail focus and notification accept both establish contact intent before join.
3. Separate low-cost global warming from selected-contact warming so detail UI can feel instant without starting heavy media/transport work for every contact.
4. Keep prewarm idempotent and observable, but avoid turning high-volume diagnostics back on by default.
