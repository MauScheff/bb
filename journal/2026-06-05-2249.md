# Journal 2026-06-05 22:49

## Summary

Turbo now has a modular network jitter buffer and audio playout engine boundary that can run legacy, Swift NetEQ-style, or shadow comparison modes without changing the app/session ownership model. The durable design rule is: transport admission validates epoch, lease, device, and packet shape; playout owns jitter, reorder, duplicate, late packet policy, FEC/PLC recovery, target delay, and resync.

## Problem

- Packet media smoothness depends on decisions that were previously spread across transport ingress, app freshness gates, adaptive buffering, Opus decode, and AVAudio scheduling.
- Fast Relay and Direct QUIC packet lanes could be made worse by deleting "late" packets before the media core had a chance to reorder, conceal, or recover them.
- We wanted RTC-grade behavior without importing an RTC session stack, and without making a risky one-way switch from the current adaptive implementation.
- The concrete failure language was: packet audio needed NetEQ-like buffering, but the implementation still needed Swift-native, algebraic, swappable, and testable boundaries.

## Design formulation

- Owner: `VoicePlayoutEngine` owns packet buffering and playout decisions after a packet has been validated into `VoicePacketV1`.
- Boundary: `PCMWebSocketMediaSession` owns Opus decode, FEC decode, PLC generation, and AVAudio scheduling effects; the playout engine owns when those effects are requested.
- Algebra:
  - `VoicePlayoutPhase`: `idle`, `buffering`, `playing`, `draining`
  - `VoicePacketAdmission`: `accepted`, `duplicate`, `late`, `wrongEpoch`, `malformed`
  - `VoicePlayoutDecision`: `hold`, `playReceived`, `playFEC`, `playPLC`, `expand`, `preemptiveExpand`, `accelerate`, `resync`
- Rollout modes:
  - `legacy-adaptive`: current adaptive scheduler is authoritative.
  - `swift-neteq-v1`: Swift NetEQ-inspired scheduler is authoritative.
  - `shadow-legacy-scheduled`: legacy schedules audible playback; Swift NetEQ observes the same validated packets and records divergence.
- Safety rule: shadow mode must not perturb audible playback. It compares frame index, recovery type, duplicate/late/missing counts, FEC/PLC counts, and resync counts while returning only legacy output.

## What changed

- Added `VoicePacketV1` as the binary Opus packet contract with `TVP1` magic, sample timestamp, frame index, flags, bounded payload length, and strict decode rejection.
- Added binary packet encode/decode at `VoiceAudioFramePayloadCodec`, so `.binaryOpusV1` can round-trip through the current payload boundary and not be misclassified as legacy PCM.
- Added `VoicePlayoutEngine` and metrics, including `shadowDivergenceCount`.
- Implemented `LegacyAdaptivePlayoutEngine` as an adapter over the existing `AdaptiveVoicePlayoutBuffer`.
- Implemented `SwiftNetEqPlayoutEngine` with:
  - bounded packet buffer
  - startup buffering gate
  - duplicate and late drop accounting
  - FEC recovery for immediately previous missing frames
  - PLC fallback
  - large-gap resync
  - jitter-smoothed target delay growth
  - buffer depth, concealment, FEC, and resync metrics
- Implemented `ShadowLegacyScheduledPlayoutEngine` for rollout comparison.
- Wired `PCMWebSocketMediaSession` to use `VoicePlayoutEngineFactory.make(mode:)`.
- Threaded `VoiceMediaCoreMode` through `makeDefaultMediaSession` and live session creation.
- Added debug/live controls:
  - `TurboDebugLiveVoiceMediaCoreMode`
  - `-TurboDebugVoiceMediaCoreMode <legacy|shadow|swift-neteq-v1>`
  - `TurboDebugBinaryVoicePacketV1Enabled`
- Documented the architecture in `docs/client/VOICE_MEDIA_CORE.md` and added proof routing in `TESTING.md`.

## What worked

- The new playout engine boundary let us make the NetEQ-like behavior independent from the old adaptive buffer while preserving the existing AVAudio scheduling surface.
- Shadow mode gave us a low-risk test cell: the new engine sees real packet timing without becoming audible.
- Keeping Opus decode, FEC decode, and PLC as closures preserved effect ownership and avoided hidden decoder state inside the scheduler.
- Binary packet validation moved malformed packet rejection to the boundary, so downstream playout sees a typed packet instead of loose metadata.
- Focused Swift tests were enough to prove the packet codec, mode selection, shadow conformance, jitter target-delay adaptation, and receive-session wiring before any physical-device retest.

## What not to repeat

- Do not drop packet media at app ingress merely because local queue delay looks high. Packet lateness belongs to playout after epoch and lease validation.
- Do not make SwiftNetEq the default just because it compiles. Default stays `legacy-adaptive`; first real traffic should run shadow, then targeted authoritative tests.
- Do not mix the semantic rollout cells:
  - playout engine mode
  - binary packet advertisement
  - raw byte datagram transport
  - encryption
  - Apple voice-processing behavior
- Do not call the current string payload boundary "true raw binary datagrams." `VoicePacketV1` is binary-framed, but Direct QUIC/Fast Relay still need a later transport-boundary change to carry raw bytes end-to-end.
