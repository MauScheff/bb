# Journal 2026-06-02 20:43

## Summary

Simulator scenario fuzzing found two selected-conversation projection bugs after the runtime/tooling path was made replayable. The durable lesson is that replayable diagnostics should become narrow app projection regressions first, then broad reliability gates should prove the fix did not only satisfy one minimized seed.

## Problem

- A minimized simulator fuzz replay preserved stale Device PTT / wake-ready state after backend transmit authority was lost while the selected conversation was waiting for the peer.
- The same replay later exposed `selected.pending_beep_dominates_live_projection`: accepting an incoming Beep could briefly project `waitingForPeer` while the backend request was still pending.
- The first concrete artifact was `/tmp/turbo-scenario-fuzz/20260602-201018-72468c2d/seed-20260603`.

## Design formulation

- Selected conversation projection is app-owned when backend truth, local Device PTT continuity, pending actions, and relationship evidence disagree.
- A selected conversation with connected Device PTT evidence but no backend transmit authority must tear down into a disconnecting projection when there is no local restore barrier exemption or remote transmit/audio evidence.
- An accepted incoming Beep must keep the incoming-Beep projection dominant until the app has actually moved into the local pending-join phase; backend request submission alone is not equivalent to joining.
- The right shape is a projection invariant plus reducer-style regressions, because the backend state was coherent and the bug lived at the app boundary where multiple evidence streams are joined.

## What changed

- Fixed selected-conversation projection in `ConversationDomain.swift` for lost transmit authority while waiting for the peer.
- Added a restore-barrier guard so recent explicit/system leaves do not project stale wake-ready state.
- Fixed pending incoming-Beep projection so `requestingBackend` does not suppress the Beep state before local pending join begins.
- Fixed the simulator scenario runtime config default path so replays exercise the intended runtime configuration.
- Added regressions for stale wake-ready projection, lost transmit authority teardown, and accepted incoming-Beep backend submission.
- Fixed a Swift test compile blocker in the media-session fixture setup.

## What worked

- The minimized replay reproduced the contradiction without physical devices.
- Engine trace extract/replay passed, which kept ownership at app projection rather than engine state.
- Strict merged diagnostics went from invariant violations to clean after the projection fixes.
- Focused regressions captured the state-machine rules directly before the broad reliability gate ran.

## What not to repeat

- Do not treat a simulator fuzz failure as tooling-only until the scenario has been replayed with a valid runtime config.
- Do not patch Device PTT teardown in the effect executor when the contradiction is in selected-conversation projection.
- Do not collapse backend request submission and local pending join into the same projection phase; they carry different evidence.
