## Summary

Post-Cloudflare-proxy-off probing of the hosted Turbo control plane showed no outright HTTP failures, but repeated multi-second tail-latency outliers on simple Unison Cloud routes. The same outlier shape appears on both the DNS-only custom domain (`https://beepbeep.to`) and the direct Unison service URL (`https://mauscheff.unison-services.cloud/s/turbo-v7/`), so the remaining HTTP route variance does not look specific to Cloudflare proxying.

## Problem

Turbo's simulator and physical-device reliability loops are sensitive to control-plane tail latency. The median route latency is acceptable for development, but occasional 2-7s responses make join/readiness convergence flaky and can trip client timeouts or recovery paths.

This became visible after disabling Cloudflare proxying for `beepbeep.to`. Earlier Cloudflare proxied behavior did produce worse failures, including Cloudflare 520s, so older WebSocket disconnect observations should not be over-attributed to Unison Cloud without a fresh DNS-only WebSocket soak.

Minimal standalone Python probe:

```python
#!/usr/bin/env python3
import json, statistics, subprocess, time, urllib.parse

BASE_URLS = [
    "https://beepbeep.to",
    "https://mauscheff.unison-services.cloud/s/turbo-v7/",
]
HANDLE = "@avery"
ITERATIONS = 12
TIMEOUT_SECONDS = 8

def url(base, path):
    return urllib.parse.urljoin(base.rstrip("/") + "/", path.lstrip("/"))

def curl(base, method, path, body=None, auth=False):
    cmd = [
        "curl", "-sS", "--max-time", str(TIMEOUT_SECONDS),
        "-w", "\n__code=%{http_code} __time=%{time_total}\n",
        "-X", method,
    ]
    if auth:
        cmd += ["-H", f"x-turbo-user-handle: {HANDLE}", "-H", f"Authorization: Bearer {HANDLE}"]
    if body is not None:
        cmd += ["-H", "Content-Type: application/json", "--data-binary", json.dumps(body)]
    p = subprocess.run(cmd + [url(base, path)], text=True, capture_output=True)
    response, metrics = (p.stdout.strip().rsplit("\n", 1) if "__code=" in p.stdout else ("", ""))
    fields = dict(part.split("=", 1) for part in metrics.split() if "=" in part)
    return {
        "ok": p.returncode == 0 and fields.get("__code", "0").startswith("2"),
        "code": int(fields.get("__code", "0")),
        "ms": int(float(fields.get("__time", "0")) * 1000),
        "stderr": p.stderr.strip(),
        "preview": response[:160],
    }

for base in BASE_URLS:
    rows = []
    for i in range(1, ITERATIONS + 1):
        device = f"stew-latency-probe-{i}"
        requests = [
            ("health", "GET", "v1/health", None, False),
            ("config", "GET", "v1/config", None, False),
            ("auth", "POST", "v1/auth/session", {"deviceId": device, "deviceLabel": "stew-latency-probe"}, True),
            ("device", "POST", "v1/devices/register", {"deviceId": device, "deviceLabel": "stew-latency-probe"}, True),
            ("invites-incoming", "GET", "v1/invites/incoming", None, True),
            ("invites-outgoing", "GET", "v1/invites/outgoing", None, True),
            ("heartbeat", "POST", "v1/presence/heartbeat", {"deviceId": device}, True),
        ]
        for name, method, path, body, auth in requests:
            r = curl(base, method, path, body, auth)
            rows.append((name, r["ms"], r["ok"], r["code"], i))
            print(base, f"{i:02d}", name, r)
    print("\nSUMMARY", base)
    for name in sorted(set(r[0] for r in rows)):
        ms = [r[1] for r in rows if r[0] == name]
        failures = [r for r in rows if r[0] == name and not r[2]]
        print(name, {"failed": len(failures), "medianMs": int(statistics.median(ms)), "maxMs": max(ms)})
```

## Current Working Conclusion

The evidence does not prove that Unison Cloud is “at fault.” It proves that after removing Cloudflare proxying, the direct Unison service and DNS-only custom domain both still show multi-second tail latency on simple control-plane routes. That makes the remaining issue either hosted infrastructure variance, application/storage route variance, or an interaction between them.

The next useful measurement is to split total request time into:

- app route execution time, emitted by the backend
- platform/routing time, inferred from client total time minus backend execution time
- WebSocket connection lifetime and frame round-trip timings, measured by a dedicated DNS-only soak

Results:

- Pure route, no auth/storage/telemetry/fork, `POST /heartbeat`: `120` requests, `0` failures, min/median/max `631ms / 707ms / 1928ms`, `0` responses >= `2000ms`.
- Route plus no-op `forkAt pool()`, `POST /heartbeat-fork`: `60` requests, `2` timeout failures, min/median/max `646ms / 754ms / 8000ms`, `2` responses >= `2000ms`.
- Route plus local `Remote.fork`, `POST /heartbeat-local-fork`: `60` requests, `5` timeout failures, min/median/max `634ms / 744ms / 8000ms`, `5` responses >= `2000ms`.
- Route plus one `OrderedTable.write`, `POST /heartbeat-storage`: `60` requests, `2` timeout failures, min/median/max `667ms / 758ms / 8000ms`, `10` responses >= `2000ms`.

Interpretation: pure hosted routing did not reproduce the multi-second tails in this sample. Adding either fork scheduling or a single durable storage write did. That narrows the most likely contributors for Turbo heartbeat and other hot control-plane routes to hosted `Remote.fork` scheduling and durable storage latency, not Cloudflare or BeepBeep-specific route code alone.
